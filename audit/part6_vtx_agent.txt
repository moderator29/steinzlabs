# PART 6: VTX AGENT AUDIT

## 6.1 ARCHITECTURE OVERVIEW

The VTX Agent is the core AI intelligence layer of Steinz Labs. It uses a dual-model strategy:
- **Executor:** claude-sonnet-4-6 — handles all requests, calls tools, processes results
- **Advisor:** claude-opus-4-6 — consulted on complex decisions via the advisor-tool-2026-03-01 beta header (max 2 uses per request by default)

This advisor strategy is documented in lib/services/anthropic.ts and is a legitimate Anthropic beta feature that delivers near-Opus quality at approximately 80-90% Sonnet cost.

## 6.2 SYSTEM PROMPT — VTX_SYSTEM_PROMPT

### Personality and Role
- Platform persona: "VTX, the intelligence layer of Steinz Labs"
- Not an assistant — positioned as a professional crypto intelligence engine
- Directives: surgical precision, actionable intelligence, no vague hedging, always structured output

### Output Format Rules
Every response must include:
- VTX Risk Score (0-100) for security-related queries
- One-paragraph AI Summary
- Recommendation: BUY / CAUTION / AVOID / NEUTRAL / INSUFFICIENT DATA

### Data Integrity Rules (Hard Constraints)
The system prompt contains explicit rules preventing hallucination:
- Never invent, estimate, or hallucinate numbers
- "Data unavailable" if field is missing or null
- Never describe portfolio value unless given exact USD figure
- Never describe trading frequency without real tx count and date range

### Archetype Classification
7 archetypes with specific rules:
- DIAMOND_HANDS, SCALPER, DEGEN, WHALE_FOLLOWER, HOLDER, INACTIVE, NEW_WALLET
- Each has specific data criteria for classification

### Assessment of System Prompt
The system prompt is well-crafted and comprehensive. The data integrity constraints are exactly right for a platform that processes real financial data. The anti-hallucination rules align with the VTX system prompt rules (see SECURITY CENTER RULES section in the prompt).

## 6.4 TOOL EXECUTION PIPELINE — lib/ai/vtxAgent.ts

### vtxQuery() — Non-Streaming
```
runVTXAgent(options):
  1. Call vtxQuery() → Anthropic API
  2. If response.stop_reason === 'tool_use':
     - Extract tool_use blocks
     - Call onToolCall(tool_name, tool_input) for each
     - Collect tool results
     - Append tool_result blocks to messages
     - Loop (max MAX_TOOL_ITERATIONS = 5)
  3. If stop_reason === 'end_turn': return final text
```

### vtxStream() — Streaming
```
streamVTXAgent(options):
  1. Call vtxStream() → returns ReadableStream<string>
  2. Stream text chunks directly to client
  NOTE: Tool use not supported in streaming mode
```

### Critical Finding — Streaming Breaks Tool Loop
When `stream: true` is set, the system uses `vtxStream()` which bypasses the tool-calling loop entirely. The streaming path returns text only — tools defined in VTX_TOOLS are never executed during streaming. This means:
- Streaming responses cannot call token_security_scan, wallet_profile, entity_lookup, etc.
- Streaming mode is limited to the model's internal knowledge only
- Any VTX chat that uses streaming cannot access real-time on-chain data

### Critical Finding — Chat Route Has No Tool Handler
In `app/api/vtx-ai/chat/route.ts`, `runVTXAgent()` is called without an `onToolCall` handler. The tool definitions are passed but the execution handler is missing:

```
runVTXAgent({
  messages,
  tools: VTX_TOOLS,
  // onToolCall is not provided
})
```

Without `onToolCall`, the agent will request tool execution but receive no results, causing it to either hallucinate data or state it cannot access the information.

The `/api/vtx-ai/route.ts` route (the main non-chat route) has proper tool execution including auto-detection of wallet addresses and token addresses from the message text.

## 6.5 RATE LIMITING

In `app/api/vtx-ai/route.ts`:
- In-memory rate limiting: 25 requests per day per IP
- Resets on server restart (same in-memory cache issue as all other features)
- No per-user rate limiting — only per-IP
- No rate limit persistence across serverless instances

This means on Vercel with multiple instances, each instance has its own rate limit counter — a user could send 25 requests to instance A and 25 more to instance B with no enforcement.

## 6.6 ADVISOR TOOL INTEGRATION

The advisor tool is configured correctly:
```
{
  type: 'advisor_20260301',
  name: 'advisor',
  model: 'claude-opus-4-6',
  max_uses: 2  (configurable via maxAdvisorUses parameter)
}
```

The `anthropic-beta: advisor-tool-2026-03-01` header is correctly set. This is passed via the second argument to `client.messages.create` using a function cast to bypass TypeScript types, which is correct for beta features not yet in the official SDK types.

## 6.7 CONTEXT MANAGEMENT

- Messages array is passed from the frontend — no server-side message history storage
- No conversation persistence in Supabase for VTX chat sessions
- If the user refreshes the page, chat history is lost
- The main VTX route (not chat) does maintain a running message array within a single request's tool loop, but not across requests

---

## 6.8 MANAGED SESSIONS API (EXPERIMENTAL)

`lib/agents/sniper-agent.ts` and `lib/agents/cluster-agent.ts` both attempt:
```
client.beta.sessions?.create?.()
```

This is the Anthropic Managed Sessions API (beta). The optional chaining (`?.`) means if the API is unavailable, the agents fall back to a standard approach. This is forward-looking code that will work when the Sessions API becomes stable but is gracefully skipped today.

---

## 6.9 ASSESSMENT: B+

VTX Agent is architecturally sound. The dual-model strategy, comprehensive system prompt, and 9 well-defined tools are production quality. The tool-calling loop is correctly implemented. The main failures are:
1. Streaming mode bypasses tool execution entirely
2. Chat route missing onToolCall handler
3. Rate limiting is in-memory only — ineffective on serverless

## 6.10 RECOMMENDATIONS

1. **CRITICAL** — Add `onToolCall` handler to app/api/vtx-ai/chat/route.ts. Without it, the chat route cannot execute any of the 9 VTX tools. Map each tool name to its corresponding service function.

2. **HIGH** — Fix streaming tool execution. Two options:
   Option A: Stream the text portion of responses while processing tool calls synchronously in the background before streaming final synthesis
   Option B: Use a two-phase approach — non-streaming tool gathering phase, then streaming synthesis phase

3. **HIGH** — Replace in-memory rate limiting with Supabase-backed rate limiting. Store request timestamps per user_id in a `rate_limits` table. This survives cold starts and works across serverless instances.

4. **HIGH** — Add conversation persistence. Store VTX chat sessions in Supabase `vtx_sessions` table per user. Allow users to resume previous sessions and use context from prior conversations.

5. **MEDIUM** — Add tool execution timeout handling. If a tool call (e.g., Arkham entity lookup) takes longer than 8 seconds, the agent should continue with available data rather than blocking the entire response.

6. **MEDIUM** — Add tool result caching. If the same wallet_profile or token_security_scan is called twice within 60 seconds, serve from cache rather than re-calling the API.

7. **MEDIUM** — Expose tool call trace in the response for transparency — let users see "VTX called token_security_scan for 0xABCD..." so they understand what data sources were consulted.

8. **LOW** — Add a `reasoning` parameter that triggers extended thinking mode in claude-opus-4-6 advisor for the most complex analysis requests.

9. **LOW** — Consider increasing max_uses for the advisor to 3 for high-complexity queries (portfolio analysis with security + market + entity lookups).

10. **LOW** — Add model override capability in VTXQueryOptions to let specific routes use full Opus for maximum quality when cost is acceptable (e.g., admin research queries).
