# PART 2: WALLET SYSTEM AUDIT

## 2.1 OVERVIEW

The Steinz Labs wallet system is a **read-only, non-custodial, browser-extension-only** architecture. The platform never holds private keys, never stores seed phrases, and never generates wallets. All signing is delegated to MetaMask (EVM) or Phantom (Solana) browser extensions installed by the user. The server stores only public wallet addresses.

---

## 2.2 WALLET CONNECTION — lib/wallet.ts

### How It Works
- Detects `window.ethereum` (MetaMask) for EVM chains
- Detects `window.solana` (Phantom) for Solana
- Calls `eth_requestAccounts` (EVM) or `connect()` (Solana) to get the connected address
- Stores `wallet_address` and `wallet_provider` in `localStorage`
- No private keys, no seed phrases, no signing keys ever touch the server

### What Is Stored in localStorage
```
wallet_address: "0x..." or base58 Solana address
wallet_provider: "metamask" | "phantom"
```

### Auto-Connect — lib/wallet/autoConnect.ts
- On page load, reads `wallet_address` from localStorage
- Checks if the browser wallet still has the same address connected
- If session is older than 48 hours, clears and requires manual reconnect
- **No server-side session persistence** — this is a purely client-side mechanism

### Cross-Device Behavior
- **No cross-device recovery exists.** If a user connects on Device A, Device B has no knowledge of the wallet. The localStorage entry does not exist on Device B.
- There is no QR code pairing, no cloud session sync, and no wallet-connect v2 integration.

---

## 2.3 WALLET ACCOUNT STORAGE — lib/wallet/walletManager.ts

### Supabase Storage
- When a wallet connects, `addWalletProfile()` inserts the address into the `wallet_accounts` table
- Columns stored: `userId`, `address`, `chain`, `label`
- This is the only server-side persistence of wallet identity

### Multiple Wallets
- FREE tier: 1 wallet maximum
- PRO tier: 3 wallets maximum
- PREMIUM tier: 10 wallets maximum
- `getUserWallets()` queries Supabase to retrieve all registered wallets for a user

### Wallet Labeling
- Users can label wallets (e.g. "Main Wallet", "Trading Wallet")
- Labels stored in Supabase `wallet_accounts` table

---

## 2.4 APPROVAL REVOCATION — CRITICAL STUB

### File: lib/wallet/walletManager.ts — revokeApproval()

**This function is a stub.** It returns:
```
{ txHash: 'pending_signature' }
```

No ERC-20 approval revocation transaction is ever built. No `approve(spender, 0)` call is constructed. The function does nothing on-chain.

**Impact:** Users who click "Revoke" in the Approval Manager UI will receive a response indicating success, but their on-chain approval remains active. This is a false sense of security — arguably the most dangerous UX failure in the security module.

**Fix Required:** Construct and return a proper `approve(spender, 0)` transaction payload using ethers.js or viem, return it as base64 for client-side signing, then submit via the connected wallet.

---

## 2.5 APPROVAL MANAGER — app/api/security/approvals/route.ts

### What Is Real
- Reads Alchemy ERC-20 `Approval` event logs for the wallet address
- For each approval, calls `allowance()` to get current remaining allowance
- Enriches each approval with token metadata (symbol, name) via Alchemy
- Runs GoPlus address security check on each spender
- Classifies known spenders against a hardcoded dictionary of 14 protocol addresses (Uniswap v2, v3, 1inch v3/v4/v5, SushiSwap, Permit2, ParaSwap, OpenSea Seaport)
- Sorts results: danger first, then warning, then safe

### What Is Incomplete
- **Revocation is a stub** (see 2.4 above)
- Only covers EVM chains — no Solana SPL token approval equivalent
- Known spender list is hardcoded in the route file — not fetched from any live source
- KNOWN_SPENDERS map covers only Ethereum mainnet addresses — polygon/base/arbitrum versions of the same protocols are not in the list

### Risk Classification Logic
- `danger`: spender is flagged malicious or phishing by GoPlus
- `warning`: allowance is unlimited (max uint256)
- `safe`: specific allowance, clean spender

---

## 2.6 WALLET REPUTATION — lib/security/walletReputation.ts

### How It Works
1. Calls `getAddressIntel()` from Arkham
2. Checks labels array for: `scammer`, `rug_puller`, `phishing` → returns DANGEROUS, score=0, allowAccess=false
3. Checks for `mixer`, `tornado_cash`, `mule` → returns SUSPICIOUS, score=15, allowAccess=false
4. If Arkham entity is verified → returns VERIFIED, score=95, allowAccess=true
5. Gets wallet connections from Arkham (first 20)
6. Counts connections to mixer/mule/scammer addresses
7. If >5 suspicious connections → SUSPICIOUS, score=25
8. Otherwise scores based on transaction count + entity presence

### Scoring Formula
```
score = min(50 + txCount * 0.1 + (arkhamEntity ? 20 : 0), 85)
```

A wallet with 100 transactions and no Arkham entity scores: min(50 + 10 + 0, 85) = 60 (UNKNOWN). A verified entity scores 95. This is reasonable but:

**Issues:**
- Score cap of 85 for non-entity wallets means even the most active legitimate wallet never scores above 85
- The `allowAccess: true` even for SUSPICIOUS wallets with <5 connections — only >5 suspicious connections triggers the flag. A wallet with exactly 5 mixer connections is allowed full access.
- Relies entirely on Arkham data. If ARKHAM_API_KEY is missing, all wallets return score=50/UNKNOWN/allowAccess=true

## 2.7 WALLET-TO-PORTFOLIO CONNECTION

**Solana path:**
```
wallet_address → lib/services/solana-intelligence.ts
  → helius.getBalance() + getTokenAccountsByOwner()
  → helius.getSolanaTokenMetaBatch() (up to 100 mints)
  → birdeye.getMultiTokenPrices()
  → dexscreener.getTokenPairs() (logos, liquidity)
  → dust filter: skip < $0.50 unless blue chip
  → whale classification: MEGA / LARGE / MEDIUM / SMALL
```

**EVM path:**
```
wallet_address → app/api/portfolio/route.ts
  → zerion.getPositions() [primary]
  → zerion.getPortfolio() [PnL, totals]
  → if Zerion fails: alchemy.getTokenBalances() [fallback, no USD prices for tokens]
```

**Key limitation:** EVM Alchemy fallback sets `change24h: 0` hardcoded — no price change data without Zerion.

## 2.8 WALLET-TO-MARKET / SWAP CONNECTION

### Swap Execution Path
```
User initiates swap (UI)
  → POST /api/trade/quote (get quote)
  → POST /api/trade/execute (build TX)
    → shadowGuardian.scanTrade() [security pre-check]
    → if Solana: jupiter.buildSwapTransaction() → returns base64 TX
    → if EVM: uniswap.buildSwapTx() → returns unsigned TX data
  → TX returned to client
  → Client signs with MetaMask/Phantom
  → Client broadcasts to network
  → Server logs fee revenue (0.15% platform fee) to swap_logs + fee_revenue tables
```

This is a **correct and secure architecture** for a non-custodial platform. The server never holds funds.

## 2.9 WALLET-TO-INTELLIGENCE CONNECTION

### DNA Analyzer
- Frontend sends wallet holdings + balance + txCount to `/api/dna-analyzer`
- Server does NOT re-fetch on-chain data independently
- Quality depends entirely on what the frontend sends
- **If the frontend sends empty or partial holdings, the AI analysis is based on nothing**

### Wallet Intelligence
- Full pipeline: Zerion (EVM) or Helius+Birdeye+DexScreener (Solana)
- GoPlus security scan run on top 5 EVM token holdings (non-blocking, does not affect response if fails)

### Wallet Clusters
- Takes holder addresses from Birdeye (Solana) or Supabase smart_money_wallets table
- Fetches SWAP and TRANSFER transactions from Helius per address (batch of 15 max)
- Runs cluster detection algorithm

## 2.11 RECOMMENDATIONS — WALLET SYSTEM

1. **IMMEDIATE — Fix revokeApproval() stub.** Build the approve(spender, 0) transaction payload using ethers.js Contract interface. Return the unsigned TX as a hex string. Let the client sign and broadcast. Log the attempt to Supabase. This is a functional security feature that currently does nothing.

2. **IMMEDIATE — Fix JWT verification in middleware.** The current base64 decode of the JWT payload checks expiry and role but does not verify the cryptographic signature. Switch to `supabase.auth.getUser(token)` in middleware or use the `jose` library with the Supabase JWT secret (`process.env.SUPABASE_JWT_SECRET`).

3. **HIGH — Add Content-Security-Policy header.** The middleware applies 5 security headers but omits CSP. Without CSP, XSS attacks can exfiltrate localStorage (wallet address, session data). Add a strict CSP restricting script-src to self.

4. **HIGH — Replace hardcoded Supabase URL in notifications/route.ts.** The string `phvewrldcdxupsnakddx.supabase.co` is hardcoded on line 127. Use `process.env.NEXT_PUBLIC_SUPABASE_URL` instead to avoid breakage on project migration.

## 3.3 WALLET CLUSTERS

### Route: app/api/wallet-clusters/route.ts

### Architecture
1. Checks Supabase `wallet_clusters` cache (20 most recent) — returns if fresh
2. If no cache or refresh=1 requested:
   - If `tokenAddress` given: fetches top 50 Birdeye holders
   - Otherwise: loads addresses from Supabase `smart_money_wallets` table (40 max)
3. For each address (batch of 15): fetches Helius SWAP txs + TRANSFER txs (20 each)
4. Builds `transfers[]` and `trades[]` arrays
5. Calls `detectCluster()` from `lib/services/cluster-detection.ts`

### Cluster Detection Algorithm — lib/services/cluster-detection.ts
6 signals with weights (total possible: 130 points):
- coordinated_buy_block: weight 28 — multiple addresses buy the same token within 5 blocks
- direct_fund_flow: weight 25 — wallet A sends funds directly to wallet B
- sequential_buy: weight 22 — addresses buy same token within 10 minutes of each other
- timing_correlation: weight 20 — Pearson correlation coefficient of trade timestamps
- shared_counterparties: weight 20 — same DEX router addresses used
- repeated_co_trading: weight 15 — addresses repeatedly appear in same token trades

Behavior classification: accumulation, distribution, pump, wash_trading, unknown
Coordination score = weighted sum / max_possible * 100

### Key Limitations
- Batch size capped at 15 addresses from Helius — for tokens with many holders, this gives limited coverage
- Helius SWAP/TRANSFER data limited to last 20 txs per address — misses historical coordination
- SOL transfer amounts in lamports divided by 1e9 (correct) but value is in native SOL, not USD — no USD conversion applied to transfers
- Only Solana clusters — no EVM cluster detection (no equivalent Helius-style enhanced tx API used for EVM)

### Assessment: B
The algorithm is technically sound. The 6-signal approach with Pearson correlation is genuinely sophisticated. The limitation is data depth: 20 txs per address is shallow for detecting subtle long-term coordination patterns.

### Recommendations
1. **HIGH** — Increase Helius transaction limit from 20 to 100 per address for deeper pattern detection
2. **HIGH** — Add EVM cluster detection using Alchemy asset transfers (already available)
3. **MEDIUM** — Add USD conversion to transfer values using live SOL price
4. **MEDIUM** — Persist cluster detection results to Supabase automatically after live detection (currently only saved if POST with admin secret)
5. **LOW** — Add a minimum coordination score threshold for reporting — clusters scoring under 25 add noise
6. **LOW** — Expose Pearson correlation coefficient in the response for transparency

### Critical Finding — Execution Is Not Implemented
The `executeLimitOrder()` function in `lib/trading/advancedOrders.ts`:

```
async function executeLimitOrder(order: any): Promise<void> {
  const supabaseAdmin = getSupabaseAdmin();

    .from('limit_orders')
    .update({ status: 'FILLED', filled_at: new Date().toISOString() })

```

**This only updates the database.** No swap transaction is built. No Jupiter or Uniswap call is made. The order is marked FILLED but nothing happens on-chain.

### Critical Finding — Price Check Is Broken
The `getCurrentPrice()` function:
```
async function getCurrentPrice(tokenAddress: string, chain: string): Promise<number> {
  return 0;
}
```

It returns 0 always. The `monitorLimitOrders()` loop compares `currentPrice <= target_price` — since currentPrice is always 0, BUY orders with any target price above 0 will always trigger immediately, and SELL orders will never trigger. The monitoring loop is effectively non-functional.

### Critical Finding — No Monitoring Loop Running
The `monitorLimitOrders()` function exists but there is no cron job, scheduled function, or background worker calling it. Even if getCurrentPrice() were fixed, no trigger exists to invoke the monitoring loop.

4. VTX AI risk assessment — blocks if AI returns RISK: HIGH or RISK: CRITICAL
5. If approved: logs to Supabase `sniper_executions` table with status 'pending'

Uses Claude Sonnet with a 5-step lifecycle:
1. Security assessment (calls token_security_scan tool)
2. Market assessment (calls token_market_data tool)

5. Result reporting

The `build_sniper_transaction` tool definition:

```
url: /api/swap/quote
```

This endpoint does not exist. The correct endpoint is `/api/trade/quote`. The AI agent's TX construction step will always fail with a 404.

2. **HIGH** — Have /api/sniper/execute return a swap transaction (base64 for Solana, unsigned hex for EVM) after all 5 checks pass, so the frontend can sign and broadcast it immediately

- Checks for known phishing brand names (metamask, uniswap, coinbase, binance, etc.)
- Checks for excessive subdomain depth (> 3 dots in hostname)

### Assessment: A
GoPlus domain scanning is a legitimate and widely-used phishing database. The heuristic fallback is genuinely useful and covers the most common phishing patterns. The URL normalization handles user input well.

### Issues
- No rate limiting on this endpoint — could be abused to check large lists of domains
- GoPlus phishing database may have coverage gaps for very new phishing sites
- No caching — same URL scanned multiple times hits GoPlus each time

### Recommendations
1. **MEDIUM** — Add 10-minute cache for domain scan results
2. **MEDIUM** — Add rate limiting (max 10 scans per IP per minute)
3. **LOW** — Add Google Safe Browsing API as a second opinion for confirmed dangerous results
4. **LOW** — Add Punycode decoding check — phishing sites often use internationalized domain names that look identical to legitimate ones

## 5.6 SHADOW GUARDIAN (PRE-TRADE SCANNER)

### Route: app/api/security/scan-trade/route.ts
### Service: lib/security/shadowGuardian.ts

### Architecture
1. Runs GoPlus security + Arkham holder scan in parallel
2. Hard block conditions:
   - GoPlus honeypot = true → BLOCKED

- No caching — every trade scan re-runs all API calls

### Recommendations

2. **MEDIUM** — Lower mixer connection threshold from >10 to >5 for stronger protection
3. **MEDIUM** — Add 5-minute cache for scan results per token address to avoid redundant API calls on rapid trading
4. **LOW** — Increase AI analysis token limit to 300 for more meaningful risk assessment
5. **LOW** — Add DexScreener liquidity as a factor: tokens with < $5k liquidity should receive automatic CAUTION regardless of other signals

## 6.4 TOOL EXECUTION PIPELINE — lib/ai/vtxAgent.ts

### vtxQuery() — Non-Streaming
```
runVTXAgent(options):
  1. Call vtxQuery() → Anthropic API
  2. If response.stop_reason === 'tool_use':
     - Extract tool_use blocks
     - Call onToolCall(tool_name, tool_input) for each
     - Collect tool results
     - Append tool_result blocks to messages
     - Loop (max MAX_TOOL_ITERATIONS = 5)
  3. If stop_reason === 'end_turn': return final text
```

### vtxStream() — Streaming
```
streamVTXAgent(options):
  1. Call vtxStream() → returns ReadableStream<string>
  2. Stream text chunks directly to client
  NOTE: Tool use not supported in streaming mode
```

### Critical Finding — Streaming Breaks Tool Loop
When `stream: true` is set, the system uses `vtxStream()` which bypasses the tool-calling loop entirely. The streaming path returns text only — tools defined in VTX_TOOLS are never executed during streaming. This means:
- Streaming responses cannot call token_security_scan, wallet_profile, entity_lookup, etc.
- Streaming mode is limited to the model's internal knowledge only
- Any VTX chat that uses streaming cannot access real-time on-chain data

### Critical Finding — Chat Route Has No Tool Handler
In `app/api/vtx-ai/chat/route.ts`, `runVTXAgent()` is called without an `onToolCall` handler. The tool definitions are passed but the execution handler is missing:

```
runVTXAgent({
  messages,
  tools: VTX_TOOLS,
  // onToolCall is not provided
})
```

Without `onToolCall`, the agent will request tool execution but receive no results, causing it to either hallucinate data or state it cannot access the information.

The `/api/vtx-ai/route.ts` route (the main non-chat route) has proper tool execution including auto-detection of wallet addresses and token addresses from the message text.

## 6.5 RATE LIMITING

In `app/api/vtx-ai/route.ts`:
- In-memory rate limiting: 25 requests per day per IP
- Resets on server restart (same in-memory cache issue as all other features)
- No per-user rate limiting — only per-IP
- No rate limit persistence across serverless instances

This means on Vercel with multiple instances, each instance has its own rate limit counter — a user could send 25 requests to instance A and 25 more to instance B with no enforcement.

1. **CRITICAL** — Add `onToolCall` handler to app/api/vtx-ai/chat/route.ts. Without it, the chat route cannot execute any of the 9 VTX tools. Map each tool name to its corresponding service function.

2. **HIGH** — Fix streaming tool execution. Two options:
   Option A: Stream the text portion of responses while processing tool calls synchronously in the background before streaming final synthesis
   Option B: Use a two-phase approach — non-streaming tool gathering phase, then streaming synthesis phase

3. **HIGH** — Replace in-memory rate limiting with Supabase-backed rate limiting. Store request timestamps per user_id in a `rate_limits` table. This survives cold starts and works across serverless instances.

6. **MEDIUM** — Add tool result caching. If the same wallet_profile or token_security_scan is called twice within 60 seconds, serve from cache rather than re-calling the API.

## 7.6 MEV PROTECTION

## 8.1 CRITICAL SEVERITY

### CRIT-01: JWT Middleware Does Not Verify Signature
**File:** middleware.ts — verifySupabaseJWT() function
**Code:**
```
const parts = token.split('.');
if (parts.length !== 3) return false;
const payload = JSON.parse(atob(parts[1]));
if (!payload.sub) return false;
if (payload.role === 'anon' || payload.role === 'service_role') return false;
if (payload.exp && payload.exp * 1000 < Date.now()) return false;
return true;
```
**Risk:** A crafted JWT with a valid structure, a real-looking sub field, a non-expired exp, and a non-anon role will pass this check even though the signature is forged. An attacker who knows the expected payload structure can bypass dashboard and admin route protection entirely.
**Fix:** Replace base64 decode with `supabase.auth.getUser(token)` which performs cryptographic signature verification against Supabase's JWT secret. Alternatively, use the `jose` library with `SUPABASE_JWT_SECRET` to verify RS256 signatures.

### CRIT-02: Limit Order Execution Is a Stub
**File:** lib/trading/advancedOrders.ts — executeLimitOrder() line ~140
**Code:**
```
async function executeLimitOrder(order: any): Promise<void> {
  await supabaseAdmin.from('limit_orders').update({ status: 'FILLED', filled_at: ... }).eq('id', order.id);
}
```
**Risk:** Users place limit orders believing they will execute automatically when price is reached. They never execute. Users may miss trade opportunities and not understand why. Financial trust failure.
**Fix:** Implement on-chain swap execution calling the same pipeline as /api/trade/execute. For non-custodial architecture: prepare the TX and push to user via WebSocket for signing.

### CRIT-03: getCurrentPrice Always Returns 0
**File:** lib/trading/advancedOrders.ts — getCurrentPrice() line ~160
**Code:**
```
async function getCurrentPrice(tokenAddress: string, chain: string): Promise<number> {
  return 0;
}
```
**Risk:** The monitorLimitOrders() loop compares currentPrice (always 0) against target_price. BUY orders trigger immediately (0 <= any positive target). SELL orders never trigger. The monitoring system is entirely broken.
**Fix:** Implement using `birdeye.getTokenPrice(mint)` for Solana or `coingecko.getContractPrice(address, chain)` for EVM.

### CRIT-04: Auto-Exit Never Executes Sell
**File:** lib/moneyRadar/monitor.ts — triggerAutoExits()
**Risk:** Users who enable auto-exit for a position expect it to automatically sell when an entity exits. The function marks the position EXITED in the DB but no sell transaction is ever built or submitted. Users may lose money expecting an exit that never happens.
**Fix:** Same as limit order execution — build swap TX and either execute via a backend signing key or push to user's active session for signing.

### CRIT-05: No Monitoring Cron Job
**File:** lib/trading/advancedOrders.ts — monitorLimitOrders()
**Risk:** Even if CRIT-02 and CRIT-03 are fixed, there is no scheduled trigger to call monitorLimitOrders(). The function exists but is never called automatically. All limit orders will remain permanently PENDING without a cron mechanism.
**Fix:** Deploy Supabase Edge Functions with `pg_cron` extension, or use Vercel Cron Jobs (Pro plan), or Upstash QStash to call the monitoring endpoint every 30-60 seconds.

## 8.2 HIGH SEVERITY

### HIGH-01: Approval Revocation Is a Stub
**File:** lib/wallet/walletManager.ts — revokeApproval()
**Code:**
```
return { txHash: 'pending_signature' }
```
**Risk:** Users click "Revoke" in the Approval Manager and receive a response suggesting success. Their ERC-20 approvals remain active on-chain. This creates a false sense of security — a critical failure for a security-focused platform.
**Fix:** Build `approve(spender, 0)` transaction using ethers.js `Interface.encodeFunctionData()`. Return as unsigned hex TX for client signing.

### HIGH-02: GoPlus simulateTransaction Missing POST Body
**File:** lib/security/goplusService.ts — simulateTransaction()
**Code:**
```
const body = { ... };  // built but never passed to fetch
const res = await fetch(url, { method: 'POST', headers });  // body missing
```
**Risk:** Transaction simulation requests to GoPlus are sent without the transaction data. GoPlus receives an empty POST body and returns a 400 or an error. The function silently catches this and returns a fallback "safe" result, meaning users doing transaction pre-screening get a false positive.
**Fix:** Add `body: JSON.stringify(body)` to the fetch options object.

### HIGH-03: Sniper Agent Calls Non-Existent Endpoint
**File:** lib/agents/sniper-agent.ts — build_sniper_transaction tool definition
**Code:**
```
url: '/api/swap/quote'  // This endpoint does not exist
```
**Risk:** The AI sniper agent's transaction construction step will always receive a 404. The agent cannot build or return a swap transaction. All sniper trade execution via the AI agent is broken.
**Fix:** Change to '/api/trade/quote' which is the correct existing endpoint.

### HIGH-04: Admin Research DELETE Password Exposed in URL
**File:** app/api/research/route.ts — DELETE handler line ~289
**Code:**
```
const password = searchParams.get('password');
```
**Risk:** The admin password is passed as a URL query parameter for DELETE requests. URLs with query parameters are logged by web servers, CDNs (Vercel edge logs), browser history, and referrer headers. The admin password will appear in logs.
**Fix:** Move password authentication to Authorization header or use the existing verifyAdminRequest() from lib/auth/adminAuth.ts.

### HIGH-05: In-Memory Rate Limiting Bypassed on Serverless
**File:** app/api/vtx-ai/route.ts — rate limiting block
**Risk:** The 25 req/day rate limit is stored in process memory. Each Vercel serverless instance has its own counter. A user can exceed the limit by having requests routed to different instances. On high-traffic deployments, this limit is effectively not enforced.
**Fix:** Store rate limit data in Supabase `rate_limits` table with timestamp array per IP/user. Check remaining count before processing request.

### HIGH-06: Context Feed Timestamps All Set to Current Time
**File:** app/api/context-feed/route.ts — recentTimestamp() called on all events
**Risk:** Events from 6+ hours ago appear as "just now" to users. This is misleading data presentation that could cause users to make trading decisions based on stale information they believe is current.
**Fix:** Use the actual block timestamp from Alchemy/Helius response (`tx.metadata.blockTimestamp` for Alchemy, `tx.timestamp` for Helius) as the event timestamp.

### HIGH-07: Network Graph Shows Mock Data as Default
**File:** app/api/network-graph/route.ts — generateMockData()
**Risk:** Users visiting the Network Graph page without connecting a wallet see hardcoded fake data (hardcoded addresses, fake volumes like $28.4M). This is presented without any label indicating it's mock data until the frontend reads `source: 'mock'` from the response. Users may interpret this as real network activity.
**Fix:** Return an empty graph with a message prompting wallet connection. Never present mock data as default state.

### HIGH-08: Missing Content-Security-Policy Header
**File:** middleware.ts — securityHeaders object
**Risk:** Without CSP, any XSS vulnerability in the application can exfiltrate localStorage data (wallet addresses, session tokens) to an attacker-controlled domain. XSS + no CSP = full session compromise.
**Fix:** Add `Content-Security-Policy: default-src 'self'; script-src 'self'; connect-src 'self' *.supabase.co *.anthropic.com; ...` header in middleware.ts securityHeaders.

### HIGH-09: Smart Money EVM Returns Pair Addresses as Wallets
**File:** app/api/smart-money/route.ts — DexScreener fallback
**Risk:** DexScreener pair contract addresses are returned labeled as "smart money wallets". These are liquidity pool contracts, not trading wallets. Users who follow or copy-trade these "wallets" would be sending funds to DEX contracts.
**Fix:** Remove the DexScreener fallback from smart-money entirely.

### HIGH-10: ETH Price Hardcoded in Two Routes
**Files:** app/api/smart-money/route.ts (ETH = $2500), app/api/whale-tracker/route.ts (ETH = $3000)
**Risk:** All USD volume calculations for EVM whales and smart money are based on wrong prices. At ETH $3000+ actual price, volumes are systematically underestimated at $2500 and overestimated relative to the $3000 figure. Financial data integrity failure.
**Fix:** Add `const ethPrice = await getTokenPrice('ethereum')` at the top of each route, with a 60-second cache. Replace hardcoded values.

## 10.3 ARCHITECTURAL ISSUES

### Issue 1: In-Memory Cache Is Fundamentally Incompatible With Serverless
Every feature in the platform uses `lib/api/cache-manager.ts` — a Map-based in-memory cache with TTLs. On Vercel serverless, every function invocation may be a cold start, instantly invalidating the cache. Multiple warm instances have separate cache maps. Rate limits on expensive APIs (GoPlus, Arkham, Birdeye) will be exceeded in production because the cache provides no protection across instances.

**Solution:** Replace with Redis (Upstash Redis has a generous free tier, $0.2/100k commands thereafter). Wrap the existing cache interface — migration is straightforward since the current cache has a clean get/set API.

### Issue 2: Advanced Orders Require a Persistent Runtime — Not Compatible With Serverless-Only
Limit order monitoring, DCA execution, and auto-exit checking all require a persistent background process running every 30-60 seconds. Next.js on Vercel does not provide this — every route invocation is ephemeral.

**Solutions:**
- Vercel Cron Jobs (Pro plan) — HTTP endpoint called on schedule
- Supabase Edge Functions with pg_cron — runs SQL or HTTP calls on schedule
- Separate Node.js worker process (Railway.app, Fly.io) — persistent, always-on

### Issue 3: EVM Smart Money Architecture Is Conceptually Wrong
The entire EVM smart money pipeline extracts wallets from transfer history rather than from a curated list of known profitable traders. This requires a fundamental rethink: maintain a Supabase table of tracked wallets with their historical performance metrics, updated by a background job.

### Issue 4: DNA Analyzer Input Trust Model
The DNA Analyzer trusts whatever holdings data the frontend sends. This is fragile. A robust architecture would have the analyzer fetch holdings server-side. This also means the DNA Analyzer quality varies dramatically based on how the frontend loaded the wallet data before calling the API.

### Issue 5: No Cross-Process Rate Limit Enforcement
GoPlus has 5,000 free queries/month. If 100 users each trigger 50 contract analyses (very possible), the free tier is exhausted. With an in-memory rate limit counter per process, there is no global enforcement. A Redis counter keyed to the API key would solve this.

## 10.6 FINAL VERDICT

**Platform Readiness for Production Launch: 6.5 / 10**

The core infrastructure (auth, database, swap, security scanning) is solid. The AI layer is well-architected. The data pipelines for Solana are production-quality.

The platform is NOT ready for production launch in its current state because:
- The advanced trading features (limit orders, stop-loss, DCA, auto-exit) are entirely non-functional — they store data but never execute
- The JWT middleware has an authentication bypass vulnerability
- The VTX chat route cannot execute any tools — the main user-facing AI has no live data access
- The approval revocation stub creates false security confidence

Fix the 5 critical issues in this report, and the platform reaches approximately 8.5/10 readiness. The foundation is genuinely strong — the gaps are specific and fixable, not architectural failures.
