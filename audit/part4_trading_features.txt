# PART 4: TRADING FEATURES AUDIT

## 4.1 SWAP

### Routes: app/api/trade/quote/route.ts and app/api/trade/execute/route.ts
### Service: lib/services/swap.ts

### Architecture

**Solana Swap Flow:**
1. Call `jupiter.getQuote()` via lite-api.jup.ag/swap/v1/quote
2. Run GoPlus security check on destination token (blocks honeypots and sell tax > 50%)
3. Call `jupiter.buildSwapTransaction()` — returns base64-serialized VersionedTransaction
4. Return base64 TX to frontend
5. Frontend signs with Phantom and broadcasts to Solana RPC
6. Server logs fee revenue: 0.15% (15 bps) to `swap_logs` and `fee_revenue` tables

**EVM Swap Flow:**
1. Call `uniswap.getQuote()` using Uniswap v3 routing on the appropriate chain
2. Run GoPlus security check on destination token
3. Build unsigned EVM transaction (to, data, value, gasLimit)
4. Return unsigned TX to frontend
5. Frontend signs with MetaMask and broadcasts

**Supported Chains (Swap):**
- Solana (Jupiter)
- Ethereum (Uniswap v3)
- Base (Uniswap v3)
- Polygon (Uniswap v3)
- Arbitrum (Uniswap v3)
- Optimism (Uniswap v3)

**Platform Fee:**
- 0.15% (15 bps) on all swaps
- Logged to `swap_logs` table: userId, tokenIn, tokenOut, amountIn, amountOut, feeUsd, chain, txHash, timestamp
- Logged to `fee_revenue` table for business analytics

### Security Pre-Check (lib/services/swap.ts)
- GoPlus honeypot check — blocks if isHoneypot=true
- Sell tax > 50% blocks trade
- Runs only on EVM chains (GoPlus Solana security has limited coverage vs EVM)

### Assessment: A-
This is the most production-complete feature in the platform. Architecture is correct and secure for a non-custodial platform. Jupiter integration is real and up-to-date (lite-api.jup.ag/swap/v1). Uniswap v3 routing is real. The 15bps fee is properly logged. Shadow Guardian pre-check adds meaningful protection.

### Issues
- No price impact warning threshold — a swap with 40% price impact proceeds without user warning
- GoPlus pre-check only runs on EVM — Solana tokens skip the security check
- No minimum liquidity check before quoting (only checked in sniper execute, not in main swap)
- 1inch is implemented (lib/trading/oneinch.ts) but does not appear to be used as a fallback when Uniswap fails

### Recommendations
1. **HIGH** — Add price impact threshold warning: return `priceImpact` in quote response and block or warn if > 5%
2. **HIGH** — Run GoPlus security check on Solana tokens too (GoPlus supports Solana via chain=solana)
3. **MEDIUM** — Add 1inch as EVM swap fallback when Uniswap v3 fails or returns worse price
4. **MEDIUM** — Add minimum liquidity threshold to quote route — refuse to quote tokens with under $5,000 liquidity
5. **LOW** — Add slippage recommendations based on token liquidity and volatility
6. **LOW** — Surface the DEX being used in the quote response (Raydium vs Orca vs Meteora for Solana; Uniswap pool vs other for EVM)

## 4.2 TRADING SUITE

### Route: app/api/trading-suite/route.ts

### Architecture
This route handles advanced order types: limit orders, stop-loss, take-profit, and DCA configurations. It stores order configurations to Supabase and references `lib/trading/advancedOrders.ts` for execution logic.

### Critical Finding — Execution Is Not Implemented
The `executeLimitOrder()` function in `lib/trading/advancedOrders.ts`:
```
async function executeLimitOrder(order: any): Promise<void> {
  const supabaseAdmin = getSupabaseAdmin();
  await supabaseAdmin
    .from('limit_orders')
    .update({ status: 'FILLED', filled_at: new Date().toISOString() })
    .eq('id', order.id);
}
```

**This only updates the database.** No swap transaction is built. No Jupiter or Uniswap call is made. The order is marked FILLED but nothing happens on-chain.

### Critical Finding — Price Check Is Broken
The `getCurrentPrice()` function:
```
async function getCurrentPrice(tokenAddress: string, chain: string): Promise<number> {
  return 0;
}
```

It returns 0 always. The `monitorLimitOrders()` loop compares `currentPrice <= target_price` — since currentPrice is always 0, BUY orders with any target price above 0 will always trigger immediately, and SELL orders will never trigger. The monitoring loop is effectively non-functional.

### Critical Finding — No Monitoring Loop Running
The `monitorLimitOrders()` function exists but there is no cron job, scheduled function, or background worker calling it. Even if getCurrentPrice() were fixed, no trigger exists to invoke the monitoring loop.

### Stop-Loss and Take-Profit
Same issue — stored in DB, never monitored, never executed.

### DCA Bots
Configuration stored with `nextBuyAt` timestamps. `calculateNextBuyTime()` is implemented correctly. But there is no periodic execution mechanism. `monitorLimitOrders()` is not called anywhere in the codebase that was reviewed.

### Assessment: F
The trading suite is a UI feature with database storage but no actual execution. Orders can be created, stored, and displayed. They will never execute on-chain. This is the most serious functional gap in the platform.

### Recommendations
1. **CRITICAL** — Implement `getCurrentPrice()` using `coingecko.getTokenPrice()` or `birdeye.getTokenPrice()` — one live API call with 30-second cache
2. **CRITICAL** — Implement `executeLimitOrder()` to call the same swap pipeline as /api/trade/execute — build the TX, sign it with a custodial key or return it to the user for manual signing
3. **CRITICAL** — Deploy a monitoring cron job: use Supabase Edge Functions with pg_cron, a Vercel Cron Job (Pro plan), or an external service (Upstash QStash) to call `monitorLimitOrders()` every 60 seconds
4. **HIGH** — Implement DCA execution: same cron mechanism should check `dca_configs` for rows where `active=true` and `next_buy_at <= now()`, execute the buy, update `spent_so_far` and `next_buy_at`
5. **HIGH** — For non-custodial execution: add a webhook/callback mechanism where the server prepares the TX and pushes it to a connected frontend session for user signing
6. **MEDIUM** — Add order expiry notifications — when a limit order expires (status EXPIRED), push a notification to the user

## 4.3 SNIPER BOT

### Routes: app/api/sniper/route.ts and app/api/sniper/execute/route.ts
### Agent: lib/agents/sniper-agent.ts

### New Token Feed — GET /api/sniper
- Calls `dexscreener.getNewPairs()` — searches DexScreener for recent pairs
- **Note:** `getNewPairs()` searches for the term "pumpfun" as a proxy — not a true new-pairs API endpoint. This means only pump.fun Solana tokens are returned, not new tokens across chains.
- Each pair gets a heuristic security score based on liquidity:
  - < $1k liquidity: score -= 40
  - < $5k: score -= 20
  - > $50k: score += 12
  - > $100k: score += 20
  - Pair age < 5 minutes: score -= 10
- Returns status: safe / risky / blocked based on score thresholds (safe >= 70, risky >= 40, blocked < 40)
- GoPlus is not called in the GET handler — only a heuristic, no real security scan

### Sniper Execute — POST /api/sniper/execute
5-Step Safety Flow:
1. GoPlus honeypot + tax scan — blocks if honeypot detected or tax > 10%
2. Holder concentration check — blocks if top 10 holders > 80%
3. Liquidity check — blocks if < $10,000
4. VTX AI risk assessment — blocks if AI returns RISK: HIGH or RISK: CRITICAL
5. If approved: logs to Supabase `sniper_executions` table with status 'pending'

**Returns:** `{ blocked: false, approved: true, steps, liquidity, pair, price }` — does NOT return a transaction. The sniper execute route only approves the trade, it does not build or submit a transaction.

### AI Sniper Agent — lib/agents/sniper-agent.ts
Uses Claude Sonnet with a 5-step lifecycle:
1. Security assessment (calls token_security_scan tool)
2. Market assessment (calls token_market_data tool)
3. Entry decision
4. TX construction (calls build_sniper_transaction tool — CALLS /api/swap/quote WHICH DOES NOT EXIST)
5. Result reporting

The `build_sniper_transaction` tool definition:
```
url: /api/swap/quote
method: POST
```

This endpoint does not exist. The correct endpoint is `/api/trade/quote`. The AI agent's TX construction step will always fail with a 404.

### Assessment: C
The safety pipeline (5-step) is genuinely good. GoPlus honeypot check + holder concentration + liquidity + AI is a solid pre-trade filter. The execution side is broken: heuristic-only token feed, non-existent quote endpoint, and no actual transaction output from the sniper execute route.

### Recommendations
1. **CRITICAL** — Fix build_sniper_transaction tool URL from /api/swap/quote to /api/trade/quote in lib/agents/sniper-agent.ts
2. **HIGH** — Have /api/sniper/execute return a swap transaction (base64 for Solana, unsigned hex for EVM) after all 5 checks pass, so the frontend can sign and broadcast it immediately
3. **HIGH** — Replace "pumpfun" search proxy in getNewPairs() with DexScreener's dedicated new pairs endpoint (if available in the API tier) or use Birdeye's new listing endpoint
4. **HIGH** — Add GoPlus scan to the GET /api/sniper new token feed — at minimum run it on the top 5 results asynchronously
5. **MEDIUM** — Add a maximum position size parameter to sniper execute — prevent users from sniping with > $X without additional confirmation
6. **LOW** — Add the AI sniper agent's reasoning output to the response so users can see why it approved or blocked
