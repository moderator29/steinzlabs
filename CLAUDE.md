# Project Rules for Claude Code Sessions

These rules apply to every Claude Code session in this repository. Read this file before making any commits.

## Git Commit Rules — STRICT

### Identity
- All commits must be authored as `moderator29 <101205446+moderator29@users.noreply.github.com>`
- If git config shows a different author, set it before committing:
  ```bash
  git config user.name "moderator29"
  git config user.email "101205446+moderator29@users.noreply.github.com"
  ```

### Forbidden in Commit Messages
Never add any of the following to commit messages:
- `Co-Authored-By: Claude <noreply@anthropic.com>` or any AI co-author trailer
- `🤖 Generated with Claude Code`
- `Generated with Claude Code`
- `Generated by Claude`
- `AI-assisted` or `AI assisted`
- Any other phrase indicating AI involvement

### Forbidden in Code, Comments, Docs, READMEs, PR Descriptions
Never add any of the following anywhere in this repository:
- "Generated by Claude" or any variant
- "AI-generated" or any variant
- "Built with Claude Code"
- Comments referencing Claude or AI assistance by name
- Auto-generated header blocks attributing AI

### Forbidden Branch Naming
Never create branches with these prefixes: `claude/`, `ai/`, `claude-code/`.
Use functional, descriptive branch names: `feat/whale-tracker-upgrade`, `fix/wallet-persistence-bug`, `refactor/api-error-handling`, `chore/repo-cleanup`, `security/<area>`.

### Branching & Merging
- Never commit to `main`. The owner (`Phantomfcalls` / `moderator29`) is the only one who merges to main; Vercel auto-deploys from main.
- Always work on a feature branch cut from main: `git checkout main && git pull --ff-only && git checkout -b <branch>`.
- Push the branch and stop — do not open the PR via `gh`. The owner opens the PR in the GitHub UI and merges.

## Commit Message Format

Use Conventional Commits:
- `feat:` new feature
- `fix:` bug fix
- `refactor:` code restructuring without behavior change
- `chore:` maintenance, dependencies, tooling
- `docs:` documentation
- `test:` tests
- `style:` formatting only
- `perf:` performance improvement
- `security:` security fix

Example:
```
feat: implement whale tracker live feed with SSE

Wires Alchemy webhooks to a Server-Sent Events endpoint so the frontend
receives whale activity in real time without polling.
```

## Code Style

- No `any` types in TypeScript unless absolutely necessary (and document why)
- No `console.log` statements in production code (use proper logging)
- No commented-out code (delete it; git remembers)
- No empty `try/catch` blocks
- All functions over 50 lines should have a brief docstring
- Match existing code style — don't introduce new patterns without reason
- Use `lib/utils/addressNormalize.ts` for all address comparisons. Never call `.toLowerCase()` directly on a wallet/token address — Solana is case-sensitive.

## Mock Data — Forbidden
This is a production trading platform. Never introduce mock/fake/demo data, hardcoded sample wallets, or stubbed API responses. Wire to the real APIs (CoinGecko, Alchemy, Helius, GoPlus, Jupiter, 0x, Anthropic, Supabase). If the data is unavailable, return an empty state with an error, never fabricated numbers.

## Documentation
- Update `/docs/` whenever a feature changes meaningfully
- Update `README.md` when adding major features or changing setup steps
- Update `CHANGELOG.md` for every release

## Security
- Never commit secrets, API keys, or credentials. The repo's `.gitignore` blocks `.env*` and `.vercel/`; do not work around it.
- Never log sensitive data (private keys, seed phrases, passwords, raw JWTs). Sentry's `beforeSend` already scrubs cookies — extend it if you add new sensitive paths.
- Server-side validate everything; never trust client-side validation alone.
- For Supabase work: prefer `apply_migration` via MCP and mirror the SQL into `supabase/migrations/` for repo parity. Never bypass RLS via `service_role` from a client-callable endpoint without explicit user-id binding.
- Reference `SECURITY.md` for the full security policy.

## Supabase Schema Gotchas
- `whales.label` (not `name`)
- `price_alerts.price` (not `target_price`)
- `user_wallets_v2.wallets` JSONB — addresses live inside the JSON, with `default_address` separate
- Verify column existence against the live DB via `mcp__supabase__list_tables` or `execute_sql`. Migration files can be stale — multiple past audits flagged false-positive "missing column" claims by reading old migrations.
